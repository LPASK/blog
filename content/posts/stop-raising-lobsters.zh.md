你兴致勃勃地给自己的 agent 系统加了定时任务。过了一阵子，Claude 官方出了 `/loop`[^1]。

过去几个月，我给自己做了一大堆东西。打通 Telegram 集成，设计了多层记忆机制，搭了 Manager-Worker 结构按需组装 agent 运行环境。还试过用 Pi-mono 从头写 agent。

每一样都很有意思。每一样都让我学到东西。但做到后面发现，复杂度本身成了负担——维护自己造的轮子比用它解决问题花的时间还多。而且回头看，大部分精力花在了社区或官方迟早会给出更好方案的地方。

这不是第一次了。我之前做过几年游戏引擎开发，踩过一模一样的坑。

虚幻引擎有一套程序化内容生成的框架——简单说就是用规则自动生成游戏场景，不用手工一个个摆。黑客帝国那个 demo 出来之后，很多团队开始自研管线。我们做了不同的选择：最大程度贴合官方设计，最小化自造轮子，只做必要的 bug 修复和功能扩展，积极给官方反馈。官方代表的是整个生态演进的方向，用一个小组去对抗生态没有意义，顺着走才是事半功倍。

Agent 时代也是一样。Claude Code 就是我的龙虾——我每天用它工作十几个小时，它是所有 agent 工作流的底座。记忆管理、IM 集成、定时任务、多 agent 编排——这些我花了几个月做的事情，社区和官方都在做。你一个人花几个月打磨出来的东西，可能下个月就有了更成熟的版本。但如果你没做过，那个更好的方案出来你也判断不了它好在哪。

现在 Agent 生态非常开放，变化很快——这不是说什么都别碰。你得动手做，才能理解设计意图，才能判断什么适合自己。只是对个人来说，花很多时间打磨是不必要的。把市面上的东西看懂，动手实现一下确认自己理解正确，自己能用起来就行了。

现在我本地在用的所有东西都非常简单，全部是 Claude Code 的扩展。记忆就是文本文件，启动时读当天的 markdown 恢复上下文，hook 防止上下文不完整导致的信息遗漏。IM 就是一个 Telegram bot 连着固定脚本。定时任务就是 shell 加 launchd。没有 embedding，没有向量检索，没有多端同步。够用。所有能力都沉淀成固定脚本[^2]。

从游戏引擎到 agent 系统，教训是同一个：个人不要重复造轮子。官方和社区有大量资源在做这些事情，个人精力不可能比过他们。

但有一件事只有你自己能做：让 AI 更懂你。你的工作习惯、决策偏好、项目上下文、犯过的错——这些东西没有任何官方或社区方案能替你积累。我现在每天结束时让 Claude 总结当天的 session，写进 memory 文件。下次启动它就知道昨天聊到哪、哪些决定做过了、哪些坑踩过了。用了几周之后，很多事情不用重新解释，它直接接着上次的上下文往下走。这就是复利——不需要 fancy 的功能设计，但需要持续投入。

所以不是什么都不做。是不养龙虾，但要养自己——养数据、养上下文、养和 AI 的默契。

Claude Code 就是最好的龙虾。别自己养了，骑在上面跑。
