**[agent-assembly](https://github.com/LPASK/agent-assembly)** — 5 个 bash 函数，~140 行代码。这是我每天在用的方案，解决的是使用 Claude Code 跨项目进行[复利工程（Compound Engineering）](https://github.com/EveryInc/compound-engineering-plugin)的痛点。

复利工程的核心思路是：你和 agent 协作得越久，agent 对你的理解越深，给出的判断和质疑越准。但有一类信息天然是跨项目的：你的技术背景、认知模式、决策偏好、当前目标。这些东西属于"你"而不是属于某个项目。一换项目，这些全丢了。System Prompt 越塞越多就膨胀失控，不塞又会被遗忘。

这篇讲的是这个答案怎么来的——走了哪些弯路，砍掉了什么，留下了什么。

---

## 第一阶段：一体化

跟大多数人的想法一样，最初是把所有东西放在一个项目里。

我的知识库项目——skills、个人画像、认知模式、记忆系统、feed 处理流程，全堆进配置。Profile 越写越细，行为规则越加越多，System Prompt 攒到了上千行。

实际用起来撑不住。因为所有东西都在一个项目里，使用过程中不断积累新的规则和 prompt——System Prompt 越来越长、越来越臃肿，agent 的指令遵从能力持续下降。写了上千行规则，该触发的不触发，该遵守的不遵守。加上 hook 不断往上下文里灌记忆、agent 自己也不断读文件，非 System Prompt 的内容在长对话中被压缩丢失，agent 基于残缺信息做出错误判断。

这就是"单一仓库"的问题。所有东西堆在一起，只会越来越膨胀，直到不可用。

## 第一个想法：用 Plugin 一步到位

单一仓库不行，那就拆出来。最初的设想很简单：做一个 Plugin，装到每个项目里，自动注入个人 profile。一个 Plugin 解决所有问题。

试了之后发现，Plugin 机制注入不了 System Prompt。

这才是关键。为什么一定要进 System Prompt？因为这里面藏着一个两难：

- **放进 System Prompt**：内容不会被压缩、不会遗忘，但 System Prompt 越长指令遵从越差，而且使用过程中只会越积越多。
- **不放 System Prompt**：System Prompt 保持精简，但内容在长对话中会被压缩丢失，模型随时可能遗忘。

全塞进去不行，全不塞也不行。需要一个机制来控制**什么该进 System Prompt、什么不该进**——而且这个决策不能交给运行时的模型，得在启动前就确定好。

Plugin 机制做不到这件事。得换一个思路。

## 第二阶段：拆分与踩坑

于是我拆了。分出一个独立项目做 Manager——专门管"我是谁"，让每个项目的 agent 保持单一职责。思路是：Manager 控制哪些 profile 模块注入哪个项目的 System Prompt，解决前面那个两难。

方向对了，但实现过程中踩了一堆坑。

**Hook**。System Prompt 里写的规则 agent 不一定遵守（Phase 1 的教训），那能不能用 hook 做硬性保障？比如写一个 Stop hook：退出时检查 agent 是否写了今天的记忆，没写就拦截。逻辑很合理——但 hook 的拦截消息触发 agent 回复，回复之后又尝试退出，又触发 hook，同一条消息循环了 60 次。问题出在哪？没有充分测试。每次手动拼配置，这次对了下次可能不对，不稳定就会出这种问题。验证好的东西就应该固化成稳定的脚本——这也直接引出了最终方案的思路。

**配置合并**。Manager 要往每个项目注入 hook 配置，但项目本身已经有自己的配置文件。两份配置怎么合到一起？一开始的方案是每次启动时做运行时 merge——深度合并 JSON、检测文件损坏、处理冲突。后来意识到完全搞反了：如果每次启动都要合并，就意味着每次都可能出冲突。正确思路是第一次设置时走一遍"编译"，把所有冲突解决完，之后每次启动只跑固定脚本。一次配置，永久运行。

**过度抽象**。system prompt 放什么、startup 读什么、hook 注入什么——这些问题一个列表就够了。但我设计了四层上下文模型和各种 Registry 来管理它们。给简单的事情套复杂的框架，只会增加维护成本。

每砍一个"精心设计"的东西，系统反而变好用了。

## 第三阶段：为什么不自己做一个 Agent？

还走了另一条路：自己开发一个独立的 Manager Agent。折腾了一个多星期的 [Pi](https://github.com/badlogic/pi-mono)，初上手很容易，甚至很爽，但深入细节里坑特别多——自己管 Provider、处理上下文拼接、做模型调用，大量时间花在解决基础设施问题上，而这些问题 Claude Code 早就解决了。后来放弃了，还是用标准框架最好。

这个经验也让我意识到，在现有成熟框架内构建确实能省掉很多不必要的麻烦：

1. **每个人用的 Agent CLI 不一样**。需要用户自己配 Provider 和反复切换，门槛就高了。
2. **Claude Code 直接用 Claude 订阅**。不需要配 API key，是目前最简单获得 Opus 的方式。
3. **维护成本**。在 Claude Code 生态内实现，新功能搭顺风车就有了；自己做就得一个个追。

## 最终：5 个 bash 函数

绕了一大圈，回到起点：既然要用成熟框架，那就在框架上面做组装就好了。不需要自己造轮子，只需要一个启动前的"编译"步骤——把你需要的模块拼好，然后交给框架去跑。

**agent-assembly** — 5 个 bash 函数，~140 行。

```
assemble_modules  → 选好的 profile 模块拼成 CLAUDE.local.md 注入 System Prompt
assemble_hooks    → 生成 hook 配置（直接覆写，不 merge）
assemble_skills   → 软链 skill 到项目目录
ensure_gitignore  → 维护 gitignore
launch            → 启动
```

每个项目一个十几行的 launcher 脚本。选好模块——技术背景、认知模式、目标、记忆——写进脚本，运行，完事。写代码的项目组装技术背景和认知模式；知识库项目组装阅读兴趣和目标。每个 agent 只看到它需要的那部分你。

这就是对那个两难的回答：不是全塞也不是全不塞，而是**每个项目只组装它需要的模块进 System Prompt**，让 System Prompt 既有关键 profile 又保持精简。

而且因为只是启动脚本，使用场景完全由你决定：想要交互式对话窗口？想要后台静默执行任务？想用手机远程遥控？只要把需求讲出来，让模型帮你写 launcher 就行。模块按需拼接，启动方式随你定义。

---

三周下来的心得：

1. **简胜于繁**。上千行行为规则效果反而比精简版差。复杂度越高你越掌控不了——KISS principle ("Keep It Simple, Stupid")，在 agent 时代依然成立。
2. **该自主的给模型自主**。很多事情不需要你手动做，交给模型判断就好。你只需要做好两件事：积累好模块，提供好上下文。让模型自己判断你需要哪些模块，帮你组装。
3. **该固定的就固定**。测试成功的东西要长期稳定化，最好的办法就是写成固定的代码和脚本。在执行上给模型留空间，但该固定的地方写成代码。

这个东西是我现在每天自己在用的方案。如果你也被类似的问题困扰，可以让你的 agent 看一下这个库，参考一下这个思路：https://github.com/LPASK/agent-assembly/releases/tag/v0.1.0
